# Release

This is a **role-skill** — a Layer 2 artifact. The text below is written in the imperative for the **loop-agent** that runs it cycle after cycle, not for the human scaffolding the loop now. Drop it into the repo's `skills/` directory and adapt every `<…>` placeholder to the project's real release tooling before the loop ever runs it.

Releasing is **human-owned**. The loop prepares a release and presents it; a human approves what ships. This skill stops short of the irreversible step (publish / merge) unless the repo's `AGENTS.md` has explicitly raised this stage's autonomy. Default behavior is to open a release PR and hand the decision to a person.

## Purpose

Turn a green branch into a release, with every step gated. A release is the most expensive thing a loop can get wrong — a bad publish reaches users and a bad tag is permanent. So: **never release on a red gate**, and never let the loop be the only thing that decided to ship.

This skill is the release stage of the loop's six-part spine (trigger → discovery → planning → execution → **verification + this** → state-update). It assumes the verification gate already ran and passed this cycle; it re-confirms that fact rather than trusting it.

## When To Use

Run this stage only when **both** hold:

- The loop has reached a release point — a backlog item, schedule, or trigger says "cut a release" (e.g. a `release/*` branch, a `next-release` milestone closing, or a manual `/loop` invocation aimed at release).
- The verification gate is **green this cycle**: tests, lint, typecheck, and build all pass. If you have not personally re-run the gate in this stage (step 1), you do not know it is green — a green gate from an earlier cycle is stale.

Do **not** run this stage to "see if a release would work," to release a branch you have not gated, or to bump versions for their own sake. If the gate is red, stop and report; releasing is off the table until it is green.

## Process (adapt to the repo)

Each step is a runnable command that exits 0/1. `<placeholders>` are filled from the repo's real tooling during scaffolding — replace them; do not run them literally.

### 1. Confirm the gate is green

Re-run the full verification gate now. Do not trust a prior cycle's result.

- Generic: `<test-cmd> && <lint-cmd> && <typecheck-cmd> && <build-cmd>` must all exit 0.
- Claude Code: run the project's gate command(s); a hook or `/goal` done-condition may already encode this.
- Codex: run the same commands via the project's configured task runner.

If any command exits non-zero, **STOP**. Record the failure in `agent-state/failed-attempts.md` and `agent-state/loop-state.md`, do not touch the changelog, version, or any tag, and return control. A red gate ends the release this cycle — there is no override here.

Also confirm the working tree is clean and you are on the intended release ref:

```
git status --porcelain   # must be empty
git rev-parse --abbrev-ref HEAD   # must match the release branch/ref
```

A dirty tree means uncommitted work would ship untracked — STOP and report.

### 2. Update CHANGELOG and bump version (per the repo's convention)

Edit the human-authored changelog and bump the version using the repo's real mechanism. Do **not** invent a versioning scheme — use what the repo already does.

- CHANGELOG: move the unreleased entries under a new `<version>` heading, dated per the repo's format. `<placeholder: the repo's changelog file and section convention — e.g. Keep a Changelog "Unreleased" → version section>`.
- Version bump command (pick the one this repo uses; the others are noise):
  - Node: `<npm version <major|minor|patch> --no-git-tag-version>`
  - Python: `<bump tool, e.g. hatch version <part> / bump-my-version bump <part>>`
  - Rust: `<cargo set-version / cargo release version <part>>`
  - Other: `<the repo's documented bump command>`
- Determine the bump size (`major`/`minor`/`patch`) from the repo's convention — conventional-commit history, a milestone label, or an explicit instruction in the triage item. `<placeholder: how this repo decides the bump>`. If the size is ambiguous, STOP and ask the human; guessing a version is an expensive-to-reverse mistake.

**Never hand-edit generated artifacts.** Lockfiles, generated version constants, `dist/`, compiled changelogs, or anything a tool produces must be regenerated by re-running that tool — not patched by hand. If a generated file is wrong, fix the source and regenerate.

Commit the changelog + version bump as one commit, on the release branch, smallest diff possible:

```
git add <changelog-file> <version-file(s)>
git commit -m "release: <version>"
```

### 3. Run the full suite once more

The version bump and changelog edit changed the tree. Re-run the **entire** gate against the post-bump state — a passing pre-bump gate does not certify the commit you just made.

```
<test-cmd> && <lint-cmd> && <typecheck-cmd> && <build-cmd>
```

If anything now fails, **STOP**: the release commit is bad. Record it in `agent-state/failed-attempts.md`, leave the tag unwritten, and return control. Do not tag a tree whose gate you have not just seen pass.

### 4. Tag the release

Create the release tag using the repo's tag convention. A tag is effectively permanent once pushed — only create it after step 3 is green.

```
git tag -a <tag-format, e.g. v<version>> -m "release: <version>"
```

- `<placeholder: the repo's exact tag format — v-prefixed? component-scoped like pkg-name@<version>? annotated vs lightweight?>`.
- Do **not** push the tag yet. Pushing a tag often triggers a publish pipeline; pushing belongs to the human-approved step below.

### 5. Publish OR open a release PR (default: PR)

This is the human-owned decision. **Default to opening a release PR** and stopping there. Only publish directly if `AGENTS.md` explicitly grants this stage publish autonomy for this repo — and even then, publishing requires that steps 1–4 all passed this cycle.

**Default path — open a release PR for human approval:**

- Push the release branch (not the tag): `git push <remote> <release-branch>`.
- Open the PR against the integration branch:
  - Claude Code / generic with `gh`: `gh pr create --base <integration-branch> --head <release-branch> --title "release: <version>" --body <release-notes-from-changelog>`.
  - Codex / no `gh`: use the host's PR primitive or print the compare URL `<remote-compare-url>` for the human to open it.
- The PR body is the changelog section for `<version>` plus the gate evidence (which commands ran, their exit status, the test count). State plainly: *this release awaits human review; the tag is created locally and the publish has not run.*
- Stop. The human reviews and merges. Merging (and any tag push / publish pipeline that merge triggers) is theirs.

**Publish path — only when AGENTS.md explicitly allows it:**

- Push the tag: `git push <remote> <tag>`.
- Run the publish command: `<the repo's real publish step — e.g. npm publish / cargo publish / a release workflow dispatch>`.
- Verify the publish landed (`<command that confirms the artifact is live>` exits 0). If it does not, STOP and report — a half-published release is a human escalation, never a silent retry.

## Rules

- **Never skip the gate to ship.** No flag, no deadline, no "it passed last cycle" justifies tagging or publishing on an unverified or red tree. A red gate ends the release this cycle.
- **Never hand-edit generated artifacts.** Lockfiles, compiled outputs, generated version constants, and tool-built changelogs are regenerated by re-running their tool, never patched by hand. Wrong output → fix the source, regenerate.
- **A release PR is reviewed by a human before merge.** The loop opens the PR; it does not merge it, does not approve it, and does not push the tag on the human's behalf. Publishing is the human's call unless `AGENTS.md` explicitly says otherwise for this stage.
- **One version, one changelog entry, one tag per release.** No re-tagging, no force-pushing a tag, no editing a published version. A mistake after publish is a new patch release, not an in-place fix.
- **Record the release in `agent-state/`.** On success (PR opened or publish confirmed), append to `agent-state/completed.md`: the version, the release commit SHA, the tag, and whether it is awaiting-merge or published. Append to `agent-state/decisions.md`: the bump size chosen and why, and the publish-vs-PR path taken and why. On any STOP, record the reason in `agent-state/failed-attempts.md` and update `agent-state/loop-state.md` so the next cycle does not re-attempt a release that already failed its gate.
- **(MemBerry)** If MemBerry tools exist, after a successful release `berry_store` the release decision — version, what shipped, the bump rationale — linked to the project and release entities, and update the `project_state` core block with the new released version. Skip this entire rule if no MemBerry tools are present; the `agent-state/` records above are the required mechanism and are sufficient on their own.
